# Скиллы проекта

В репозитории есть два вспомогательных скилла Claude Code (`.claude/skills/`).
Оба — обёртки вокруг готового бинаря `.build/hsi_compressor`: они прогоняют
компрессор по сценариям и собирают статистику. Перед запуском проект должен быть
собран (см. [USAGE.md](USAGE.md#сборка)).

| Скилл | Назначение |
|-------|-----------|
| [`bench-compression`](#bench-compression--бенчмарк-сжатия) | Замер времени, памяти, числа эталонов и веса файлов по всем методам |
| [`refining-sc`](#refining-sc--уточнение-коэффициента-sc) | Подбор коэффициента `sc`, минимизирующего число под-эталонов |

Триггерятся скиллы фразами на русском (см. поле `description` в их `SKILL.md`),
например «прогнать бенчмарк по all.dat» или «уточнить sc для метода AT».

## `bench-compression` — бенчмарк сжатия

Файлы: [.claude/skills/bench-compression/SKILL.md](../.claude/skills/bench-compression/SKILL.md),
[bench_runs.py](../.claude/skills/bench-compression/bench_runs.py).

### Что делает

Прогоняет `hsi_compressor` на полном кубе `.data/all.dat` для каждого метода КЭКМ
и обоих режимов сдвига (`-S 0` и `-S 1`) с **заданным пользователем** `sc` на
метод. Итог — текстовая таблица сравнения.

### Что меряется

- **Время работы** и **пиковая память** — из обёртки `/usr/bin/time -v`
  (`Elapsed wall clock time`, `Maximum resident set size`).
- **Внутренних эталонов** — из stdout компрессора (строка `Внутренних эталонов: N`).
- **Вес `all.dat`** — размер входного файла (константа).
- **Вес файлов сжатия** — `standarts.gsd` + `image.gsd` прогона.

### Прогоны

8 прогонов = 4 метода × shift ∈ {0, 1}, **последовательно по одному** (чистые
замеры без конкуренции за CPU/RAM). Каждый — с `-w 1` (нужны файлы сжатия) и
`-r 0` (RGB не нужен); пути GSD разведены по `runs_bench/<METHOD>_S<shift>/`.
Размеры берутся из `hsi.hdr` (1032×8955×224, byte order 1).

Пороги `me/ae` по умолчанию: NT/ST — `--me 0.8 --ae 0.2`, OT/AT — `--me 1.6 --ae 0.4`
(те же, что в `refining-sc`). Переопределяются флагами `--me-<метод>` /
`--ae-<метод>`.

### Как запустить

Из корня репозитория. `sc` задаётся для каждого прогоняемого метода:

```bash
# все четыре метода
python3 .claude/skills/bench-compression/bench_runs.py \
    --sc-nt 1.128 --sc-ot 1.066 --sc-st 1.0 --sc-at 1.6

# только часть методов
python3 .claude/skills/bench-compression/bench_runs.py \
    --methods OT AT --sc-ot 1.066 --sc-at 1.6

# с переопределением порогов me/ae для метода
python3 .claude/skills/bench-compression/bench_runs.py \
    --methods NT --sc-nt 1.128 --me-nt 1.0 --ae-nt 0.3
```

### Выход

- `runs_bench/bench_results.txt` — итоговая таблица (строка на прогон):
  `Метод | Shift | Время | Память | Внутр.эталонов | all.dat | Файлы сжатия`.
- `runs_bench/<METHOD>_S<shift>/output.txt` — сырой вывод прогона (с отчётом
  `/usr/bin/time -v`).
- `runs_bench/<METHOD>_S<shift>/{standarts,image}.gsd` — файлы сжатия прогона.

> Полный куб ~4.14 ГБ × 8 прогонов последовательно — это долго (минуты–десятки
> минут на прогон) и ёмко по диску.

## `refining-sc` — уточнение коэффициента sc

Файлы: [.claude/skills/refining-sc/SKILL.md](../.claude/skills/refining-sc/SKILL.md),
[refine_sc.py](../.claude/skills/refining-sc/refine_sc.py).

### Что делает

Подбирает коэффициент шага решётки `sc` (см.
[MATH.md](MATH.md#геометрия-покрытия-и-смещение-эталона)), при котором число
**внутренних эталонов** минимально. Прогоняет компрессор по сетке кандидатов
`sc` **параллельно**, парсит `Внутренних эталонов: N` и накапливает статистику —
позволяя уточнять `sc` по разрядам.

### Тег прогона

Чтобы прогоны разных изображений и порогов не затирали друг друга, всё именуется
по **тегу** `<image>_<me>_<ae>` — стем файла `--dat` плюс базовые пороги
(например `hsi759_479_0.8_0.2`). По нему формируются папки результатов и файл
статистики.

### Алгоритм

1. Скрипт прогоняет компрессор для кандидатов `sc` параллельно, сохраняет вывод в
   `runs/<image>_<me>_<ae>/<METHOD>/output_<sc>.txt` (+ сводка `refined.txt`),
   где `<METHOD>` ∈ NT/OT/ST/AT.
2. Результаты добавляются в **единый** файл статистики
   `results_top_internal_<image>_<me>_<ae>.txt` (секции по всем методам). Он
   подаётся обратно через `--results` на следующей итерации уточнения.

Если файла статистики ещё нет, скрипт создаёт **шаблон**: по одной seed-строке на
метод с `sc = 1.0` и заведомо огромным числом эталонов — любой реальный прогон её
побьёт, так что уточнение стартует от `sc = 1.0`.

### Как запустить

```bash
# дефолтное изображение и пороги
python3 .claude/skills/refining-sc/refine_sc.py

# другое изображение с его размерами
python3 .claude/skills/refining-sc/refine_sc.py \
    --hdr .data/other.hdr --dat .data/other.gsd -s 1024 -l 512 -b 0
```

### Параметры скрипта

| Флаг | По умолчанию | Назначение |
|------|--------------|-----------|
| `--methods` | `NT OT ST AT` | какие методы уточнять |
| `--results` | по тегу прогона | файл статистики (`results_top_internal_<image>_<me>_<ae>.txt`) |
| `--workers` | число ядер CPU | число параллельных процессов компрессора |
| `--hdr` | `.data/hsi.hdr` | путь к ENVI-заголовку |
| `--dat` | `.data/hsi759_479.gsd` | путь к бинарному изображению |
| `-s, --samples` | `760` | число столбцов |
| `-l, --lines` | `480` | число строк |
| `-b, --byte-order` | `0` | порядок байт (0=LE, 1=BE) |
| `-e, --me` | `0.8` | базовый порог 1-го уровня %, NT/ST |
| `-a, --ae` | `0.2` | базовый порог 2-го уровня %, NT/ST |

Базовые `me`/`ae` относятся к NT/ST; **OT/AT автоматически берут их удвоенными**
(`1.6`/`0.4` при дефолте).
